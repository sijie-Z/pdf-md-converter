## 作业背景

我们内部经常需要把 PDF 文档转换成 Markdown 或结构化中间结果，再交给智能体、RAG、检索或人工复核使用。真实工作不是简单“把 PDF 丢给模型”，而是要考虑：

- PDF 是否有文字层？是否需要 OCR？
- 标题、正文、表格、页眉页脚、脚注、图片说明是否处理正确？
- 转成 Markdown 后，是否保留页码、标题、表格和来源信息？
- 后续用于 RAG / 检索 / 人工复核时，内容是否可追溯到原 PDF？
- 哪些步骤可以交给智能体，哪些必须人工复核？
- 如何把这套流程沉淀成下次可复用的 Skill / SOP，减少重复人工工作？

本作业希望你做一个最小可运行的 PDF 转 Markdown 小系统，并设计一个可复用的 Skill / SOP。

## 任务一：实现一个最小 PDF 转 Markdown 系统

请实现一个命令行或脚本工具，输入 PDF，输出 Markdown 和结构化结果。

### 最低要求

你的程序至少能：

1. 读取 `sample_pdf_to_markdown_note.pdf`。
2. 生成 `outputs/document.md`。
3. 在 Markdown 中保留页码，例如：

```md
<!-- page: 1 -->
# 附注36 租赁负债
```

4. 尽量识别标题、正文、表格。
5. 对图片扫描页或无文字层页面做出标记
6. 生成一个结构化文件，例如 `outputs/blocks.json` 或 `outputs/tables.json`，记录页面、文本块、表格、来源位置等信息。
7. 生成 `outputs/qa_report.md`，说明解析质量、发现的问题、哪些地方需要人工复核。

### 加分但不强制

- 能使用 PyMuPDF / pdfplumber / OCR / 版面分析工具抽取表格。
- 能把表格输出为 Markdown table。
- 能识别扫描页并尝试 OCR。
- 能保留表格标题、页码、行列信息、source locator。
- 能检查合计数是否等于明细项之和。

### 不接受的做法

- 手工把 PDF 内容复制成 Markdown，绕过解析流程。
- 只输出纯文本，不保留页码、表格、扫描页提示或质量报告。
- 只写方案不运行。

## 任务二：沉淀一个 PDF 转 Markdown 质量检查 Skill / SOP

请在仓库中新增：

```text
skills/pdf_to_markdown_review_skill.md
```

这个 Skill 的目标是：让智能体以后拿到 PDF 和 PDF 转 Markdown 结果时，能先做一轮解析质量检查、风险标记和人工复核提示，减少人工重复检查工作。

请至少包含：

1. Skill 名称和适用场景。
2. 不适用场景或必须人工介入的场景。
3. 输入文件要求：PDF、Markdown、结构化 blocks/tables、解析日志等。
4. 输出格式：质量报告、问题清单、需人工复核清单、优化建议。
5. 执行步骤：如何检查页码、标题、正文、表格、图片页、扫描页、来源追溯。
6. 质量检查 checklist：页码、标题层级、表格完整性、行列结构、数值、单位、脚注、扫描页、合计数、source locator。
7. 常见错误和处理方式：页眉污染、表格错位、跨页表格、OCR 低置信、乱码、空页、图片说明缺失等。
8. 人工复核点：哪些情况不能自动判断。
9. 如何持续优化：新增 bad case 后怎么更新 Skill，怎么让下次检查更省人工。
10. 一个小示例：输入一页 Markdown 或一条表格解析结果，输出质量判断和复核建议。

### 我们希望看到的能力

- 不是只写一段 Prompt，而是写成团队可复用的流程。
- 能说明 Skill 如何减少人工工作量，但不会过度自动化到“无需人工”。
- 能体现你愿意钻研优化：例如如何处理扫描页、跨页表格、表格标题、数值格式、页码来源和 bad case 回写。

## 我们会重点看什么

1. 你的系统是否能真实运行，而不是只写设计。
2. Markdown 是否保留页码、标题、表格、扫描页标记和来源。
3. 你是否能生成结构化中间结果，方便后续质量检查和复核。
4. 你是否知道哪些内容必须人工复核，尤其是扫描页、表格、关键数值和来源不完整的内容。
5. Skill 是否可复用、可维护、能减少人工重复劳动。
6. 是否愿意钻研优化，而不是只交一个“能跑一下”的脚本。
7. AI 使用是否具体，有没有复核意识。
8. README 和 Git 协作习惯是否清楚。
