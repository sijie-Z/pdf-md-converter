# PDF to Markdown 质量检查 Skill / SOP

> **文档版本**: 1.0  
> **适用团队**: 数据工程 / 内容管线 / AI 标注审核  
> **最后更新**: 2026-06-13

---

## 1. Skill 名称和适用场景

**名称**: `pdf_to_markdown_review`

**适用场景**:

- 团队内部需要把 PDF 文档（财报、研报、合同、招股书等）批量转成 Markdown 用于 RAG / 检索 / 人工复核。
- 智能体或管线工具（如本仓库的 `convert_pdf.py`）已完成初步解析，需要自动化质量检查。
- 人工复核前先过一遍自动检查，缩小人工复核范围。
- 版本迭代后回归测试：改了解析逻辑或换了 OCR 引擎后，用此 Skill 跑一遍确认没有退化。

**典型工作流**:

```
PDF 文档
    ↓
[解析管线] → 产出 document.md + blocks.json
    ↓
[本 Skill] → 产出 qa_report.md + 问题清单
    ↓
[人工复核] → 仅检查高风险的 flagged 项
    ↓
[反馈修正] → 更新 bad case 库 → 优化解析管线
```

## 2. 不适用场景 / 必须人工介入的场景

### 不适用场景

| 场景 | 原因 |
|------|------|
| PDF 被加密 / 有密码保护 | 无法读取内容，需先解密 |
| PDF 全为纯图片且无 OCR 引擎可用 | 无法提取文本 |
| PDF 为扫描件且质量极低（模糊、倾斜、手写） | 自动 OCR 难以保证准确率 |
| PDF 包含复杂数学公式或化学结构式 | Markdown 无法准确表达 |
| PDF 为多栏混合排版的杂志/报纸 | 文本流顺序通常混乱 |

### 必须人工介入的场景

| 场景 | 风险等级 | 原因 |
|------|---------|------|
| 关键财务数据（金额、百分比、日期） | 🔴 高 | 自动解析可能产生数值位移 |
| 表格合计行的数值一致性检查 | 🔴 高 | 合计 ≠ 明细时无法自动判断优先级 |
| 跨页表格的拼接 | 🟡 中 | 自动拼接可能错位 |
| OCR 结果中的关键数值 | 🟡 中 | OCR 识别数值的置信度通常低于文本 |
| 水印文字与正文的区分 | 🟡 中 | 水印可能被误判为正文 |
| 脚注与正文的对应关系 | 🟡 中 | 自动关联可能出错 |
| 图片说明文字的提取 | 🟢 低 | 不影响主体内容，但建议核对 |

## 3. 输入文件要求

### 必要输入

| 文件 | 格式 | 说明 |
|------|------|------|
| 原始 PDF | `.pdf` | 源文档，用于交叉验证 |
| Markdown 结果 | `.md` | `document.md`，包含页码标记、标题、表格、图片页标记 |
| 结构化中间结果 | `.json` | `blocks.json`，包含每个文本块/表格的 page、type、text、source bbox |
| 解析日志 | `.log` / 控制台输出 | 记录解析过程中的 warning / error |

### 可选输入

| 文件 | 格式 | 说明 |
|------|------|------|
| OCR 结果 | `.json` / `.txt` | 扫描页的 OCR 输出，用于与 blocks.json 交叉验证 |
| 前次 QA 报告 | `.md` | 用于对比 Regression，确认新版本是否引入新问题 |
| Bad Case 库 | `.json` | 过往记录的问题 case，用于检查当前 PDF 是否有类似情况 |

### 文件组织示例

```text
outputs/
  document.md        # Markdown 输出
  blocks.json         # 结构化块
  qa_report.md       # 质量报告（本 Skill 产出）
  qa_report_v2.md    # 新版本回归比对
```

## 4. 输出格式

### 质量报告 (`qa_report.md`)

包含以下章节：

```markdown
# PDF Parse QA Report

## Summary
总页数 / 文本页 / 图片页 / 标题数 / 表格数 / 问题数

## Per-Page Breakdown
每页的块类型和问题数

## Issues Found
- 问题类型（page_number, type, severity, detail）
- 严重级别：🔴 error / 🟡 warning / 🔵 info

## Human Review Checklist
- 哪些页需要人工看
- 看什么（数值、表格结构、OCR 结果）
- 重点关注事项

## Recommendations
优化建议：OCR、数值校验、表格调整等
```

### 问题清单 (`issues.json`)

结构化格式，便于 CI/CD 集成：

```json
[
  {
    "page": 3,
    "type": "image_page_no_ocr",
    "severity": "warning",
    "detail": "Page detected as image-only; no OCR applied."
  }
]
```

### 人工复核清单（可用 Excel / 在线表格）

| 页码 | 需要复核 | 复核内容 | 是否完成 | 备注 |
|------|---------|---------|---------|------|
| 1 | 否 | — | — | — |
| 3 | 是 | OCR 识别结果 / 手动转录 | ⬜ | — |

## 5. 执行步骤

### Step 1: 前置检查

```
[1.1] 确认 PDF 可读、未加密、有文字层或已 OCR 预处理
[1.2] 确认 document.md 存在且非空
[1.3] 确认 blocks.json 存在且非空
[1.4] 读取解析日志，记录出现的 warning / error
```

### Step 2: 页码完整性检查

```
[2.1] 扫描 document.md，提取所有 `<!-- page: N -->` 标记
[2.2] 检查页码数量是否与 PDF 实际页数一致
[2.3] 检查页码是否连续（无跳号、无重复）
[2.4] 标记缺失或重复的页码
```

### Step 3: 标题层级检查

```
[3.1] 从 blocks.json 中提取所有 type=heading 的块
[3.2] 检查标题是否按原文顺序出现
[3.3] 检查是否有 heading 被误判为 paragraph，反之亦然
[3.4] 检查标题文本是否完整（无截断、无乱码）
```

### Step 4: 正文完整性检查

```
[4.1] 检查每个文本页的正文内容是否被完整提取
[4.2] 检查正文中是否有异常断行（不该换行的地方换行了）
[4.3] 检查是否有乱码字符（如 □、�、乱码）
[4.4] 检查页眉/页脚/页码是否被正确过滤（不应出现在正文中）
```

### Step 5: 表格检查

```
[5.1] 从 blocks.json 中提取所有 type=table 的块
[5.2] 检查表格行数和列数是否与原始 PDF 一致
[5.3] 检查表头（header）是否提取正确
[5.4] 检查数值单元格是否有值位移（值跑到了错误的列）
[5.5] 检查合并单元格的处理情况
[5.6] 检查表格标题（表 N-N）是否存在于 blocks.json
[5.7] 检查表格脚注是否保留
```

### Step 6: 数值检查

```
[6.1] 如果表格有 合计（total）行，计算明细之和 vs 合计值
[6.2] 检查数值格式是否一致（小数位数、千分位分隔符）
[6.3] 检查百分比值是否保留 % 符号
[6.4] 检查单位（百万元/万元/元）是否在脚注或表头保留
```

### Step 7: 图片 / 扫描页检查

```
[7.1] 从 blocks.json 中提取所有 type=image_page 的块
[7.2] 确认 scan / image 页面被正确标记
[7.3] 如果已做 OCR，检查 OCR 文本与原文的匹配度
[7.4] 如果未做 OCR，标记为 NEEDS_OCR_REVIEW
```

### Step 8: 来源追溯检查

```
[8.1] 随机抽取 2-3 个 block，检查 bbox（page, x0, y0, x1, y1）是否在 PDF 中对应
[8.2] 确认 source.method 字段记录了使用的解析引擎
[8.3] 确认每个 block 都有 page 字段
```

### Step 9: 汇总报告

```
[9.1] 汇总所有 issues，按 severity 排序
[9.2] 生成 Human Review Checklist
[9.3] 生成优化建议
[9.4] 输出 qa_report.md
```

## 6. 质量检查 Checklist

### 通用

- [ ] PDF 页数是否全部覆盖（无遗漏页）
- [ ] 页码标记是否连续、无跳号
- [ ] 输出文件（document.md, blocks.json）是否生成
- [ ] 是否有解析 warning / error（检查日志）

### 标题

- [ ] 所有标题是否按正确层级保留（### → ## → # 的对应关系）
- [ ] 标题是否被错误地当作 paragraph 处理
- [ ] 标题文本是否完整、无乱码

### 正文

- [ ] 正文是否按原始阅读顺序排列
- [ ] 是否有重复段落（页眉污染）
- [ ] 是否有段落被错误丢弃
- [ ] 是否有乱码或不可打印字符

### 表格

- [ ] 表头是否正确提取
- [ ] 行数是否正确（无缺行、无多余行）
- [ ] 列数是否正确（无缺列、无多余空列）
- [ ] 数值是否在正确的列中
- [ ] 合计行：明细之和 vs 合计值（误差 < 0.01）
- [ ] 表格标题是否保留
- [ ] 表格脚注是否保留
- [ ] 合并单元格是否处理（标注未处理即可）

### 数值

- [ ] 数字格式是否保留（负数、百分比、小数点）
- [ ] 单位是否保留（在表头或脚注中）
- [ ] 合计值是否等于明细之和

### 脚注

- [ ] 脚注文本是否保留
- [ ] 脚注是否关联到正确的表格/段落

### 扫描页 / 图片页

- [ ] 扫描页是否被标记为 needs_ocr
- [ ] 图片说明是否提取
- [ ] OCR 结果（如有）是否经过人工复核

### Source Locator

- [ ] 每个 block 是否有 page 字段
- [ ] 每个 block 是否有 bbox 或来源描述信息
- [ ] 来源信息是否可用于追溯到原始 PDF

## 7. 常见错误和处理方式

| 错误类型 | 现象 | 可能原因 | 处理方式 |
|---------|------|---------|---------|
| 页眉污染 | 每页 Markdown 开头都有重复的公司名/报告名 | header/footer 未过滤 | 更新 `likely_page_header()` 正则规则 |
| 表格错位 | 数值出现在错误的列 | 表格线识别不完整 | 检查 pdfplumber table settings；或手动标注列边界 |
| 跨页表格断裂 | 同一个表格被拆到两页，中间有 `<!-- page: N -->` 分割 | 天然跨页 | 通过 table_id 识别跨页表格，在 QA 中标记需拼接 |
| OCR 低置信度 | OCR 输出的文字有大量错误 | 图片质量低 / 字体异常 | 标记为 NEED_MANUAL_TRANSCRIBE |
| 乱码字符 | □、�、\x00 等 | 编码识别错误 / PDF 字体未嵌入 | 在 blocks.json 中标记乱码位置，人工补录 |
| 空页 | 页面无内容但仍被解析 | PDF 有空白扫描页 | 检测页面尺寸 + 无文本 → 标记为 empty_page |
| 图片说明缺失 | 图片下的说明文字未提取 | 图片与文字分离 | 检测图片相邻位置的文字块 |
| 合计数不匹配 | 合计 ≠ 明细之和 | PDF 本身不一致 / 提取错误 | 标记为 error，人工核对原始 PDF |
| 标题误判 | 正文被标为 heading 或反之 | 字体大小/样式判断不准 | 调整 font_size 阈值或添加例外规则 |
| 数值精度丢失 | 120.50 → 120.5 | 自动 trim | 在表格提取时保留原始字符串，不做 float 转换 |

## 8. 人工复核点

以下情况 **必须** 人工复核，不能依赖自动检查：

### 🔴 必须复核（高优先级）

- 扫描页且未做 OCR → 需要人工转录
- 表格合计 ≠ 明细之和 → 需要确认是 PDF 错误还是提取错误
- OCR 结果中的关键数值（金额、日期、百分比）→ 与原始图片一一核对
- 跨页表格 → 需要人工确认拼接结果

### 🟡 建议复核（中优先级）

- 列数较多的表格（≥6 列）→ 容易发生列错位
- 包含合并单元格的表格 → 自动提取通常不完全正确
- 包含特殊字符（✓、✗、●）的文档 → 可能被转为 □
- 脚注内容较多时 → 检查脚注与正文的关联关系

### 🟢 抽样复核（低优先级）

- 普通文本页 → 每 5-10 页抽检 1 页
- 格式统一的简单表格 → 每 3 个表格抽检 1 个
- 标题层级 → 随机抽检 3-5 个标题

### 人工复核工作步骤

```
1. 打开原始 PDF 和 document.md 并排对照
2. 从 "Must Review" 列表开始检查
3. 每个 checked item 记录：PASS / FAIL / NEED_FIX
4. FAIL 项记录：页码 + 问题描述 + 预期修正
5. 汇总到复核表格
```

## 9. 持续优化

### Bad Case 管理

建立 Bad Case 库（`bad_cases/` 目录），每个 case 记录：

```json
{
  "id": "bc_001",
  "title": "表头被误判为正文",
  "pdf": "sample_report_2025.pdf",
  "page": 12,
  "description": "表头行文字 font-size=10.5，低于阈值 11，被分为 paragraph",
  "root_cause": "HEADING_FONT_SIZE_MIN 阈值过高",
  "fix": "降低阈值至 10.0，或添加表头行位置检测",
  "fixed_in_version": "v1.2",
  "added_at": "2026-06-01",
  "status": "fixed"
}
```

### 优化工作流

```
发现 Bad Case
    ↓
记录到 bad_cases/ 目录
    ↓
分析根因（阈值问题？规则遗漏？引擎限制？）
    ↓
在 convert_pdf.py 中修复 → 更新版本号
    ↓
用修复后的版本重新解析该 PDF → 确认 fix 有效
    ↓
回归测试：用历史 PDF 跑一遍，确保无退化
    ↓
更新 docs / comments
```

### 如何减少人工工作量

| 措施 | 效果 | 难度 |
|------|------|------|
| 增加 font-size 检测精度（按字体聚合） | 减少标题误判 | 低 |
| 增加表格 bbox 缓存（避免重复全页查找） | 加速表格检测 | 低 |
| 增加合计数自动校验 | 自动发现数值问题 | 中 |
| 记录 Bad Case 并定期回顾 | 持续提升准确率 | 低 |
| 对 OCR 结果做置信度过滤（低于阈值则标记） | 减少低质量 OCR 的误导 | 中 |
| 跨页表格拼接 | 减少人工整理工作 | 高 |
| 使用版面分析模型（LayoutLM / DocTR） | 大幅提升版面理解 | 高 |

### 版本管理

```text
bad_cases/
  README.md
  bc_001_table_caption.md
  bc_002_footer_leak.md
versions/
  v1.0_initial.md
  v1.1_fix_table.md
```

每次修改解析逻辑后：
1. 更新 `convert_pdf.py` 版本号
2. 在 `versions/` 中记录变更
3. 更新本 Skill 文档的版本号和最后更新日期

## 10. 小示例

### 输入

从 `document.md` 截取的一页：

```markdown
<!-- page: 1 -->

## 附注36 租赁负债

以下数据用于测试 PDF 转 Markdown、表格抽取、页码来源保留和 GT 候选审核。

**表36-1 租赁负债到期分析**

<!-- t1 -->
| 项目 | 2026-06-30 | 2025-12-31 |
| --- | --- | --- |
| 一年以内 | 120.50 | 110.25 |
| 一至两年 | 88.00 | 91.00 |
| 两至五年 | 160.75 | 149.30 |
| 合计 | 369.25 | 350.55 |

_注：金额单位为百万元；合计数应等于各明细项目之和。_
```

对应 `blocks.json` 节选：

```json
{
  "page": 1,
  "block_id": "tbl_1",
  "type": "table",
  "header": ["项目", "2026-06-30", "2025-12-31"],
  "rows": [
    ["一年以内", "120.50", "110.25"],
    ["一至两年", "88.00", "91.00"],
    ["两至五年", "160.75", "149.30"],
    ["合计", "369.25", "350.55"]
  ]
}
```

### 质量检查输出

```markdown
## QA 判断: ✅ PASS (需要抽样复核)

### 检查结果

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 页码标记 | ✅ | `<!-- page: 1 -->` 存在 |
| 标题层级 | ✅ | `## 附注36 租赁负债` 正确 |
| 正文提取 | ✅ | 描述段落完整 |
| 表格标题 | ✅ | `**表36-1 租赁负债到期分析**` 保留 |
| 表格行数 | ✅ | 5 行（表头 + 4 数据行） |
| 表格列数 | ✅ | 3 列（项目、2026-06-30、2025-12-31） |
| 合计一致性 | ✅ | 2026-06-30: 120.50+88.00+160.75=369.25 ✓ |
| 合计一致性 | ✅ | 2025-12-31: 110.25+91.00+149.30=350.55 ✓ |
| 脚注保留 | ✅ | `_注：金额单位为百万元；..._` 保留 |
| Source Locator | ✅ | bbox 信息在 blocks.json 中 |

### 复核建议

- **无需人工复核**: 该页所有数值合计一致，表格结构完整
- **抽样验证**: 如有余力，可抽查一列值是否与原始 PDF 一致
- **注意**: 金额单位为百万元，在 RAG 使用时应保留单位信息
```

### 假设发现问题的输出

```markdown
## QA 判断: ❌ FAIL (需人工复核)

### 检查结果

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 合计一致性 | ❌ | 2026-06-30: 合计=369.25, 明细之和=365.50, 差额=3.75 |
| Source Locator | ⚠️ | 缺少 bbox 信息 |

### 复核建议

- **必须人工复核**: 合计行数值不一致，需查看 PDF 原始数据
  - 可能是提取漏行（缺少某明细项）
  - 可能是合计值提取错误
  - 也可能是 PDF 本身四舍五入差异

- **操作步骤**:
  1. 打开原始 PDF 第 1 页
  2. 逐行核对 2026-06-30 列数值
  3. 确认合计值 = 369.25 还是明细之和 = 365.50
  4. 如确认为提取错误，更新 blocks.json 后重新生成 document.md
```

---

> **本 Skill 的宗旨**: 自动做能做的事，明确标记不能做的事，让人工复核只做必须做的事。
>
> 每次迭代的目标不是消除人工复核，而是让每一次人工复核都用在最有价值的地方。
